Foreword and acknowledgments

This is a text intended for a course on operating systems. It explains the main concepts of operating systems by studying an example kernel, named logOS. The name is a portmanteau of "Logisim OS", reflecting the fact that logOS was originally designed to run on a simulated RISC-V computer built in the Logisim digital logic simulator. The name also evokes the Greek word logos (λόγος), meaning "word", "reason", or "principle" — a fitting allusion for a system whose purpose is to lay bare the fundamental principles of how an operating system works, stripping away the complexity of production systems to reveal the essential logic beneath.

logOS is modeled on the Unix operating system designed by Ken Thompson and Dennis Ritchie. It provides a subset of the services and system calls that Unix kernels traditionally offer: processes, file descriptors, a filesystem with directories, and a shell. The system is intentionally small — the entire kernel is under 3,000 lines of C and assembly — making it feasible for a student to read and understand every line of code in the course of a semester.

This text should be read alongside the source code for logOS. The approach is inspired by John Lions' Commentary on UNIX 6th Edition, which demonstrated the extraordinary pedagogical value of studying a complete, working operating system line by line. Where the xv6 teaching kernel (developed at MIT) serves this purpose for a full-featured Unix on multi-core RISC-V hardware, logOS occupies a simpler niche: a single-core, bare-metal system with no virtual memory, no preemptive scheduling, and no locking — a kernel reduced to its essence.

The simplicity of logOS is deliberate. By running on a CPU simulated in Logisim, students can observe the hardware and software in a single, integrated environment. There is no MMU, so the concepts of address spaces and memory protection are replaced by the simpler (and historically older) approach of fixed memory partitions. There is no preemptive multitasking, so the complexities of concurrent scheduling and locking vanish. What remains is a clear illustration of the core mechanisms: how a system call enters the kernel, how the filesystem translates paths to data on disk, how a shell spawns programs, and how a process model gives each program the illusion of having the machine to itself.

The target audience is second-year Computer Science students taking an introductory operating systems module. The text assumes familiarity with C programming and a basic understanding of computer architecture, particularly the concepts of registers, memory, and instruction execution. RISC-V-specific details are introduced as they arise.

Each chapter follows a consistent structure. It begins with a conceptual introduction explaining why the mechanism exists and what problem it solves. It then walks through the actual logOS source code, function by function. Each chapter concludes with a "Real world" section that compares the logOS approach to production systems like Linux, and an "Exercises" section with questions and implementation tasks.

Chapter 1: Operating system interfaces

The job of an operating system is to share a computer among multiple programs and to provide a more useful set of services than the hardware alone supports. An operating system manages and abstracts the low-level hardware, so that, for example, a user program need not concern itself with which block of disk stores a particular file. An operating system arranges for multiple programs to run, and provides controlled ways for them to interact and share data.

An operating system provides services to user programs through an interface. Designing a good interface turns out to be difficult. On the one hand, we would like the interface to be simple and narrow because that makes it easier to get the implementation right. On the other hand, we may be tempted to offer many sophisticated features to applications. The trick is to design interfaces that rely on a few mechanisms that can be combined to provide much generality.

This book uses a single operating system as a concrete example to illustrate operating system concepts. That operating system, logOS, provides a subset of the interfaces introduced by Ken Thompson and Dennis Ritchie's Unix operating system. Unix provides a narrow interface whose mechanisms combine well, offering a surprising degree of generality. This interface has been so successful that modern operating systems — BSD, Linux, macOS, and even, to a lesser extent, Microsoft Windows — have Unix-like interfaces. Understanding logOS is a good start toward understanding any of these systems.

logOS takes the traditional form of a kernel, a special program that provides services to running programs. Each running program, called a process, has memory containing instructions, data, and a stack. The instructions implement the program's computation. The data are the variables on which the computation acts. The stack organizes the program's procedure calls. A given computer typically has many processes but only a single kernel.

When a process needs to invoke a kernel service, it invokes a system call, one of the calls in the operating system's interface. The system call enters the kernel; the kernel performs the service and returns. Thus a process alternates between executing in user space and kernel space.

The collection of system calls that the logOS kernel provides is the interface that user programs see. Figure 1.1 lists all of logOS's system calls.

The rest of this chapter outlines logOS's services — processes, file descriptors, the filesystem, and the shell — and illustrates them with code snippets and discussions of how the shell uses them. The shell, logOS's command-line user interface, is an ordinary user program that reads commands from the user and executes them; the fact that the shell is a user program, and not part of the kernel, illustrates the power of the system call interface.

1.1 Processes and spawn

A logOS process consists of user-space memory (instructions, data, and stack) plus per-process state private to the kernel (a trap frame, file descriptors, environment variables, and a process identifier or "PID").

logOS provides the spawn() system call to create a new process. It loads an executable from the filesystem into a fresh memory slot, sets up its stack, and begins executing it. The calling process is suspended until the child terminates; when the child calls exit(), the parent resumes and spawn() returns the child's exit code.

This is different from Unix's fork()/exec() pair. In Unix, fork() creates a copy of the calling process, and exec() replaces the copy's memory with a new program. logOS combines these into a single spawn() call, which is simpler and avoids the need to copy the parent's entire address space. The trade-off is that logOS cannot support the Unix pattern of having the child modify its file descriptors between fork() and exec(), which is how Unix implements I/O redirection and pipelines.

Here is an example user program (user/hello.c): 

#include "libc.h"

int main(int argc, char *argv[])
{
    puts("Hello from user program!\n");
    printf("main() is at address 0x%x\n", (unsigned int)main);
    printf("argc = %d\n", argc);

    for (int i = 0; i < argc; i++) {
        printf("argv[%d] = \"%s\"\n", i, argv[i]);
    }

    return 42;
}

The program prints a greeting, its arguments, and returns exit code 42. The return from main() is converted to an exit(42) system call by the user C runtime startup code (user/user_crt0.S). This startup code is the first thing that runs when a program starts: 

_start:
    jal  main           # Call main(argc, argv)
    li   a7, 0          # SYS_exit = 0
    ecall               # Exit with main's return value in a0
1:  j    1b             # Should never reach here

The kernel passes argc in register a0 and argv in a1, following the RISC-V calling convention. After main() returns (its return value left in a0), _start issues the exit system call.

A program that wants to run another program uses spawn(). For example (user/spawn_demo.c): 

int main(int argc, char *argv[])
{
    char *child_argv[4];      /* Build argv array at runtime */
    child_argv[0] = "hello";
    child_argv[1] = "arg1";
    child_argv[2] = "arg2";
    child_argv[3] = (char *)0;

    int result = spawn("/bin/hello", child_argv);
    printf("spawn returned %d\n", result);

    return 0;
}

Note that the child_argv array is built at runtime rather than using a static initializer. This is necessary because logOS user programs are position-independent executables (PIE) loaded at arbitrary addresses. Static pointer arrays would contain absolute addresses baked in at link time, which would be incorrect after relocation. Chapter 7 explains this in detail.

1.2 File descriptors and I/O

A file descriptor is a small integer representing a kernel-managed object that a process can read from or write to. A process obtains a file descriptor by opening a file or device. logOS pre-opens three file descriptors for every process:

fdNameDescription
0Standard input (stdin)Console keyboard
1Standard output (stdout)Console display
2Standard error (stderr)Console display

The read and write system calls read and write bytes through file descriptors. A call to read(fd, buf, n) reads at most n bytes from the file descriptor fd, copies them into memory at address buf, and returns the number of bytes read. Each file descriptor that refers to a regular file has an associated offset. read reads data from the current offset and advances it.

The write(fd, buf, n) system call writes n bytes from buf to file descriptor fd and returns the number of bytes written. For device files such as the console, write sends characters to the display. For regular files, write writes data at the current offset.

The following code fragment (from the user library) writes a string to standard output (user/libc.c): 

void puts(const char *s) {
    write(STDOUT_FILENO, s, strlen(s));
}

A process can open additional files using the open system call: 

int fd = open("/bin/hello", 0);  /* Open a file for reading */
char buf[128];
int n = read(fd, buf, sizeof(buf));  /* Read some data */
close(fd);  /* Release the file descriptor */

logOS supports up to 8 file descriptors per process. Three are pre-allocated for the standard I/O channels, leaving 5 available for user programs.

1.3 The filesystem

The logOS filesystem provides a tree of named files and directories. At the top is the root directory, represented by the path /. Files within directories are named by pathnames such as /bin/hello, where bin is a directory and hello is a file within it.

Several system calls operate on the filesystem. mkdir creates a new directory. rmdir removes an empty directory. readdir reads the entries of a directory. mknod creates a device file, which is not a file containing data but rather a name that refers to a hardware device.

The chdir system call changes the process's current working directory. Once set, relative paths (those not starting with /) are resolved relative to the current directory: 

chdir("/bin");
int fd = open("hello", 0);  /* Opens /bin/hello */

Every directory contains two special entries: . (dot), which refers to the directory itself, and .. (dot-dot), which refers to its parent. For the root directory, .. points back to the root.

1.4 Device files

logOS represents hardware devices as files in the filesystem. A device file (also called a device node) is identified by two numbers: a major number that selects a device driver, and a minor number that the driver interprets. For example, the console is major 1, minor 0.

The mknod system call creates a device file: 

mknod("/dev/tty", 1, 0);  /* Create console device node */

Once created, the device can be opened and used with the same read and write calls as regular files. This uniformity — treating devices and files through the same interface — is one of the enduring design principles of Unix.

1.5 Environment variables

Each process in logOS has a set of environment variables: named string values stored within the kernel's process table. The setenv, getenv, and unsetenv system calls manipulate these.

Environment variables serve several purposes. The PATH variable tells the shell which directories to search for commands. The CWD variable stores the current working directory. The ? variable holds the exit status of the last program. When a process spawns a child, the child inherits a copy of the parent's environment. Changes made by the child do not affect the parent — this follows standard Unix semantics.

1.6 The shell

The logOS shell (user/shell.c) is an ordinary user program that reads commands from the console and executes them. It is not part of the kernel. The kernel simply loads /bin/sh at startup and restarts it if it exits.

The shell reads a line, expands $VAR references, parses the line into words, and either executes a built-in command or spawns an external program. The shell's main loop (simplified) is: 

for (;;) {
    puts(getenv("CWD"));
    puts("$ ");
    gets(line, MAX_CMD_LEN);

    expand_vars(line, expanded, MAX_CMD_LEN);
    argc = parse_cmd(expanded, argv, MAX_ARGS);

    if (strcmp(argv[0], "exit") == 0)
        exit(code);
    else if (strcmp(argv[0], "cd") == 0)
        chdir(argv[1]);
    else
        run_external(argc, argv);  /* Search PATH and spawn */
}

The run_external function searches the directories listed in $PATH for the command name, constructs the full path, and calls spawn(): 

$ hello                 # Runs /bin/hello (found via PATH=/bin)
$ /bin/hello            # Runs with explicit full path
$ echo $PATH            # Variable expansion: prints "/bin"
$ set FOO=bar           # Set environment variable
$ echo $FOO             # Prints "bar"
$ ls /bin               # List directory contents
$ mkdir /tmp            # Create a directory
$ cd /tmp               # Change working directory

1.7 Real world

The logOS system call interface is a simplified version of what Unix and its descendants provide. Production Unix systems have hundreds of system calls (Linux has over 400), while logOS has 16. logOS omits many foundational Unix features:

fork and exec. Unix separates process creation (fork) from program loading (exec), allowing the child to set up I/O redirection between the two calls. This enables pipelines (ls | grep foo). logOS's combined spawn is simpler but cannot support pipelines.

Pipes. Unix pipes provide a unidirectional byte stream between two processes. They are the mechanism behind the | operator in the shell. logOS does not implement pipes.

File permissions. Unix associates permissions (read, write, execute) with every file, controlling access based on user and group identity. logOS has no concept of users or permissions.

Signals. Unix uses signals to notify processes of events (e.g., Ctrl-C sends SIGINT). logOS has no signal mechanism.

Virtual memory. Modern Unix systems give each process its own virtual address space via hardware page tables. logOS runs on hardware without an MMU and instead uses fixed physical memory partitions.

Despite these omissions, logOS illustrates the fundamental Unix design principles: everything is a file, system calls are the interface between user programs and the kernel, the shell is just a user program, and the kernel manages resources on behalf of processes.

Chapter 2: Operating system organization

A key requirement for an operating system is to support several activities at once. For example, one might use spawn from Chapter 1 to start both a compiler and a text editor as processes. The operating system must arrange for isolation between the processes: if one process has a bug, it should not affect others. Complete isolation, however, is too strong, since it should be possible for processes to interact. Thus an operating system must balance multiplexing, isolation, and interaction.

This chapter provides an overview of how logOS is organized to achieve these goals. It describes the hardware platform, the kernel's structure, the memory layout, and the boot sequence that brings the system to life.

2.1 The hardware platform

logOS runs on a RISC-V processor simulated in Logisim Evolution, an educational digital logic simulator. The simulated system consists of:

  • A 32-bit RISC-V CPU supporting the RV32IM instruction set (integer arithmetic plus multiply/divide) with the Zicsr extension for control and status registers (CSRs).
  • ROM at address 0x00000000, containing the kernel code.
  • RAM starting at address 0x00100000, used for the kernel's data and process memory slots.
  • A block device at address 0x00200000, providing persistent storage via memory-mapped I/O registers.
  • A console (TTY) at address 0xFFFF0004, providing keyboard input and text output.

This is a single-core machine with no memory management unit (MMU). All memory addresses are physical addresses. There is no hardware memory protection: any instruction can access any address. Isolation between processes is maintained by convention (each process stays within its assigned memory slot) rather than by hardware enforcement.

2.2 Kernel organization

logOS is a monolithic kernel: all operating system services run in a single program that handles system calls. This contrasts with a microkernel design, where services like the filesystem run as separate user-level processes.

The ROM contains the bootloader, which is the first code to run after a CPU reset. The bootloader's job is to load the kernel binary from the filesystem on the block device into RAM at 0x00100000 and then jump to it. The bootloader places its own BSS section high in RAM (at 0x001F0000) so that loading the kernel at 0x00100000 does not overwrite the bootloader's mutable state.

Each process slot is 64KB. The slot contains the program's code (starting at the slot base), its data and BSS sections, and its stack (growing downward from near the top of the slot). The kernel reserves 256 bytes at the very top of each slot as a guard, so the stack pointer starts at slot_base + 0xFF00.

The process table data structure (kernel/process.h) records the base address and stack top for each slot: 

#define PROC_SLOT_SIZE  0x10000     /* 64KB per process */
#define PROC_MEM_START  0x00110000  /* First slot base address */

#define PROC_SLOT_BASE(n)   (PROC_MEM_START + (n) * PROC_SLOT_SIZE)
#define PROC_SLOT_STACK(n)  (PROC_SLOT_BASE(n) + PROC_SLOT_SIZE - 0x100)

There are 8 process slots (numbered 0 through 7). Slot 0 is always used for the shell. When the shell spawns a command, the command is loaded into slot 1. If that command spawns another program, it goes into slot 2, and so on.

2.4 The boot sequence

logOS uses a three-stage boot process: the bootloader (in ROM) loads the kernel from the filesystem into RAM, the kernel initialises subsystems, and then the kernel loads the shell into a process slot and transfers control to it.

2.4.1 Stage 1: The bootloader

When the RISC-V processor starts, it begins executing instructions at address 0, which is in ROM. The bootloader's startup code (boot/boot_crt0.S) runs first: 

_start:
    la sp, __stack_init       # Set stack pointer to top of RAM
    la t0, __bss_start        # Zero bootloader BSS section
    la t1, __bss_end          # (in high RAM at 0x1F0000)
1:  bge t0, t1, 2f
    sw zero, 0(t0)
    addi t0, t0, 4
    j 1b
2:
    jal boot_main             # Call C bootloader main
hang:
    j hang                    # Should never reach here

This code sets the stack pointer to the top of RAM (0x001FFFFC), zeroes the BSS section, and calls boot_main(). Note that the bootloader's BSS is placed at 0x001F0000 (high in RAM) by its linker script (boot/boot.lds). This avoids collision with the kernel binary, which will be loaded at 0x00100000.

The function boot_main() (boot/boot_main.c) reads the kernel from the filesystem and jumps to it: 

void boot_main(void)
{
    /* Step 1: Read and validate superblock */
    boot_block_read(0, &sb);
    if (sb.magic != FS_MAGIC)
        boot_halt();

    /* Step 2: Resolve /boot/kernel in the directory tree */
    boot_resolve_path("/boot/kernel", &sb, block_buf, &kernel_ino);

    /* Step 3: Read kernel inode to get file size and block list */
    /* ... read inode from inode table ... */

    /* Step 4: Load kernel data blocks into RAM */
    dst = (uint8_t *)KERNEL_LOAD_ADDR;  /* 0x00100000 */
    for (b = 0; b < DIRECT_BLOCKS && loaded < file_size; b++) {
        boot_block_read(kernel_in.blocks[b], dst);
        dst += BLOCK_SIZE;
        loaded += chunk;
    }

    /* Step 5: Jump to kernel entry point */
    void (*kernel_entry)(void) = (void (*)(void))KERNEL_LOAD_ADDR;
    kernel_entry();
}

The bootloader has its own minimal filesystem implementation (boot/boot_fs.c, boot/boot_io.c) — just enough to read the superblock, traverse directories, and copy data blocks. It does not need the full filesystem code that the kernel uses. The function boot_resolve_path() walks the directory tree starting from the root inode, looking up “boot” and then “kernel” in successive directory entries.

Once the kernel binary is fully loaded into RAM, the bootloader casts the load address to a function pointer and calls it. Execution transfers to the kernel's entry point at 0x00100000, and the bootloader never runs again.

2.4.2 Stage 2: Kernel initialisation

The kernel's entry point (kernel/crt0.S) is structurally identical to the bootloader's: it sets the stack pointer, zeroes the kernel's BSS, and calls main(): 

_start:
    la sp, __stack_init       # Initialise stack pointer
    la t0, __bss_start        # Zero the kernel BSS section
    la t1, __bss_end
1:  bge t0, t1, 2f
    sw zero, 0(t0)
    addi t0, t0, 4
    j 1b
2:
    jal main                  # Call C main() function
exit:
    j exit                    # Loop forever after main() returns

The BSS zeroing is essential because the RAM contents are undefined at startup; C requires that global variables initialised to zero actually be zero. The kernel and the bootloader share the same initial stack pointer value (0x001FFFFC), but since the bootloader has finished and will not run again, there is no conflict.

2.4.3 Stage 3: Loading the shell

The kernel's main() function (kernel/main.c) initialises all subsystems and enters the shell loop: 

int main(void)
{
    printf("Welcome to logOS Belfield 1.0!\n\n");

    /* Initialize subsystems */
    console_dev_init();   /* Register console device driver */
    proc_init();          /* Initialize process table */

    /* Mount the filesystem */
    result = fs_mount();

    /* Run shell in a loop — restarts after each shell exit */
    for (;;) {
        int slot = proc_alloc();        /* Get a free process slot */
        proc_env_init(slot);            /* Set PATH, CWD defaults */
        elf_load_at("/bin/sh", ...);    /* Load shell executable */
        setup_trap_frame(slot);         /* Configure trap handling */
        proc_fd_init(slot);             /* Open stdin/stdout/stderr */
        run_user_program(&p->tf);      /* Start the shell */

        /* Shell exited — free slot and restart */
        proc_free(slot);
    }
}

The initialisation order matters. The console device driver must be registered before any output can be produced. The process table must be initialised before any process can be allocated. The filesystem must be mounted before any file can be opened. Only after all this is done can the shell be loaded and started.

The elf_load_at() call reads the shell's ELF binary from /bin/sh in the filesystem and loads it into process slot 0 (at address 0x00110000). The kernel then sets up a trap frame with the shell's entry point and stack pointer, installs the trap handler in the machine-mode CSRs, and calls run_user_program(). This function saves the kernel's register state and executes mret to transfer control to the shell's _start address. At this point the shell is running, and the kernel is effectively suspended — it will resume only when the shell calls sys_exit.

Notice that the shell loop is infinite: when the shell exits, the kernel frees its process slot and loads a fresh copy. This is how production Unix systems work as well; the init process typically respawns getty and login shells when they exit.

2.5 Real world

logOS’s organisation is intentionally simple compared to production kernels. Linux, for example, has millions of lines of code organised into dozens of subsystems. However, the basic structure is the same: a bootloader that loads the kernel into memory, a startup sequence that initialises hardware and mounts filesystems, a trap handler that dispatches system calls, and a process table that tracks running programs.

The logOS bootloader is much simpler than production bootloaders. Systems like GRUB or U-Boot support multiple boot targets, configuration files, and hardware initialisation sequences. The logOS bootloader needs only to read one file from a known path. The most significant architectural difference is the lack of hardware memory protection. Production systems use an MMU to give each process its own virtual address space, making it impossible for one process to read or write another’s memory. logOS relies on software conventions instead. This was actually how early Unix systems worked on the PDP-7, before hardware memory protection became common.

Chapter 3: Traps and system calls

There are three kinds of event that cause the CPU to set aside ordinary execution of instructions and force a transfer of control to special kernel code. One is a system call, when a user program executes the ecall instruction to ask the kernel to do something. Another is an exception: an instruction does something illegal, such as accessing an invalid address. The third is a device interrupt, when a device signals that it needs attention.

This book uses trap as a generic term for all three situations. Typically whatever code was executing at the time of the trap will later need to resume, and shouldn’t need to be aware that anything special happened. A trap forces a transfer of control into the kernel; the kernel saves registers and other state so that execution can be resumed; the kernel executes appropriate handler code; the kernel restores the saved state and returns from the trap; and the original code resumes where it left off.

This chapter explains the trap mechanism in detail, starting with the conceptual problem it solves and then walking through the logOS implementation.

3.1 Why system calls cannot be ordinary function calls

When a user program calls a library function like strlen(), the compiler generates a jal (jump and link) instruction that transfers control directly to the function’s code. The caller and the callee share the same set of registers and the same stack. The callee can freely read the caller’s memory, and the caller can jump to any address it likes. This works fine within a single program because all of its code is trusted.

A system call is fundamentally different. When a user program asks the kernel to read a file or spawn a process, it is asking a different program — the kernel — to perform work on its behalf. The kernel has access to hardware devices, the process table, the filesystem, and other resources that user programs must not touch directly. If a user program could simply jump to a kernel function with jal, it could jump to any kernel address, corrupt kernel data structures, or bypass any checks the kernel might perform.

The solution is a controlled transfer. Rather than jumping directly to the kernel, the user program executes a special instruction — ecall on RISC-V — that asks the CPU hardware to transfer control to the kernel at a single, pre-determined entry point. The hardware guarantees that execution always begins at this fixed address (stored in the mtvec register), not at an arbitrary location chosen by the user program. The kernel code at that entry point can then examine what the user program requested and decide how to handle it.

3.2 Context and context switching

When a trap occurs, the CPU is in the middle of executing a user program. That program has values in all 32 general-purpose registers, and the program counter (pc) points to the instruction that was about to execute (or, in the case of ecall, to the instruction that caused the trap). Together, these register values form the program’s execution context — a snapshot of everything the CPU needs to resume the program exactly where it left off.

The kernel also needs to use registers. It needs a stack pointer to call C functions. It needs temporary registers to examine the trap cause and dispatch to the right handler. It needs argument registers to pass data to those handlers. But there is only one set of registers on the CPU. If the kernel simply started using them, the user program’s register values would be destroyed, and there would be no way to resume the program afterwards.

The solution is context switching: before the kernel begins its own work, it saves the user program’s entire register state to memory. When the kernel is finished, it restores the saved register state and transfers control back to the user program, which resumes as though nothing happened. The complete cycle — save the user’s context, run kernel code, restore the user’s context — is the core mechanism underlying every system call, exception, and interrupt.

The data structure used to save a program’s register state is called a trap frame. Each process in logOS has its own trap frame (stored in the process table), so the kernel can save one process’s context, run a different process, and later switch back by restoring the original trap frame. The code that performs the save and restore is called the trap handler.

It is worth pausing to appreciate the chicken-and-egg nature of this problem. The trap handler must save all registers to memory, but to store anything in memory it needs a pointer — and that pointer must be in a register. Every register it uses to perform the save is a register it has not yet saved. Different architectures solve this problem in different ways; the next section describes the RISC-V approach.

3.3 RISC-V trap machinery

The RISC-V CPU has a set of hardware control and status registers (CSRs) that govern trap handling. logOS runs in machine mode (M-mode), which is the highest privilege level and the only one that the Logisim CPU implements. The most important CSRs are:

  • mtvec: The kernel writes the address of its trap handler here. When a trap occurs, the CPU jumps to the address in mtvec.
  • mepc: When a trap occurs, the CPU saves the program counter here (since the pc is overwritten with the value from mtvec). The mret instruction copies mepc back to the pc, resuming execution.
  • mcause: The CPU writes a number here that identifies the cause of the trap. For system calls (ecall), the cause is 11.
  • mscratch: A scratch register available for the trap handler's use. logOS stores the address of the current process's trap frame here.

When a user program executes ecall, the following happens in hardware:

  1. The current pc is saved in mepc.
  2. The cause code (11 for ecall) is written to mcause.
  3. The pc is set to the address in mtvec.
  4. Execution continues at the trap handler.

Note that the hardware does not save general-purpose registers, switch stacks, or do anything else. All of that is the trap handler's responsibility.

3.4 Code: The trap handler

The assembly language trap handler (kernel/trap.S) is the first code that runs when a trap occurs. Its job is to save the complete CPU context (all 32 general-purpose registers plus the exception program counter) into a data structure called the trap frame, switch to the kernel stack, and call the C language trap handler.

The trap frame is a structure stored in the process table (kernel/trap.h): 

struct trap_frame {
    uint32_t reserved1;        /*   0 */
    uint32_t c_trap_sp;        /*   4  Kernel stack pointer */
    uint32_t c_trap;           /*   8  C trap handler address */
    uint32_t mepc;             /*  12  Exception program counter */
    uint32_t reserved3;        /*  16 */
    uint32_t ra;               /*  20  Return address */
    uint32_t sp;               /*  24  Stack pointer */
    /* ... gp, tp, t0-t2, s0-s1 ... */
    uint32_t a0;               /*  56  Argument/return registers */
    uint32_t a1, a2, a3, a4, a5, a6;
    uint32_t a7;               /*  84  System call number */
    /* ... s2-s11, t3-t6 ... */
};

The trap handler faces a chicken-and-egg problem: it needs a register to use as a pointer to the trap frame, but all registers belong to the user program and must be saved. The solution uses the mscratch CSR. Before entering user mode, the kernel stores the trap frame address in mscratch. The trap handler begins with an atomic swap: 

trap_handler:
    csrrw a0, mscratch, a0    # Swap a0 and mscratch atomically

After this instruction, a0 holds the trap frame pointer, and mscratch holds the user program's original a0 value. Now the handler can save all other registers into the trap frame: 

    sw ra,  20(a0)            # Save return address
    sw sp,  24(a0)            # Save stack pointer
    sw gp,  28(a0)            # Save global pointer
    /* ... save all other registers ... */
    sw a7,  84(a0)            # Save syscall number

    csrr t0, mepc             # Save exception program counter
    sw t0, 12(a0)

    csrr t0, mscratch         # Recover original a0 from mscratch
    sw t0, 56(a0)             # Save it in trap frame

With all user registers saved, the handler switches to the kernel stack (stored in the trap frame at offset 4) and calls the C trap handler: 

    lw sp, 4(a0)              # Switch to kernel stack
    lw t0, 8(a0)              # Load C handler address
    jalr ra, t0, 0            # Call c_trap_handler(tf)

3.5 Code: The C trap handler and syscall dispatch

The C trap handler (kernel/main.c) reads the trap cause and dispatches accordingly: 

void c_trap_handler(trap_frame_t *tf) {
    uint32_t cause = get_mcause();

    if (cause == 11) {
        /* System call (ecall) */
        if (syscall_dispatch(tf)) {
            program_should_exit = 1;
            return;
        }
        trap_ret(tf);
    } else if (cause & MCAUSE_INTERRUPT) {
        printf("Unexpected interrupt\n");
        trap_ret(tf);
    } else {
        printf("Exception! mcause=%d, mepc=0x%x\n", cause, tf->mepc);
        while (1) {}  /* Halt on exception */
    }
}

For system calls (cause 11), the handler calls syscall_dispatch (kernel/syscall.c), which reads the system call number from tf->a7 and dispatches to the appropriate handler: 

int syscall_dispatch(trap_frame_t *tf) {
    uint32_t syscall_num = tf->a7;
    int32_t result = 0;

    switch (syscall_num) {
        case SYS_exit:  return sys_exit(tf);
        case SYS_read:  result = sys_read(tf); break;
        case SYS_write: result = sys_write(tf); break;
        case SYS_open:  result = sys_open(tf); break;
        /* ... other system calls ... */
    }

    tf->a0 = (uint32_t)result;   /* Store return value */
    tf->mepc += 4;               /* Advance past ecall */
    return 0;
}

Two crucial things happen after every system call. First, the return value is placed in tf->a0, which becomes register a0 when the user program resumes — exactly where the C calling convention expects a function's return value. Second, mepc is advanced by 4 bytes (one instruction) to skip past the ecall instruction. Without this, the program would execute ecall again in an infinite loop.

3.6 Code: Returning from a trap

After the system call is handled, the kernel calls trap_ret (kernel/trap.S) to restore the user program's context: 

trap_ret:
    lw t0, 12(a0)            # Restore mepc
    csrw mepc, t0

    lw t0, 56(a0)            # Put user's a0 in mscratch
    csrw mscratch, t0

    lw ra,  20(a0)           # Restore all registers
    lw sp,  24(a0)
    /* ... restore all other registers ... */

    csrrw a0, mscratch, a0   # Atomic swap: restore a0, save tf ptr

    mret                     # Return from trap (jump to mepc)

The mret instruction copies mepc to the program counter, resuming execution at the instruction after the ecall.

3.7 Code: User-space system call stubs

On the user side, each system call has a small assembly stub (user/syscall.S) that places arguments in the correct registers and executes ecall: 

/* ssize_t read(int fd, void *buf, size_t count) */
read:
    li   a7, 1          # SYS_read = 1
    ecall               # Trap into kernel
    ret                 # Return to caller (result in a0)

The RISC-V calling convention places the first argument in a0, the second in a1, and so on. The system call stubs exploit this: the C caller already has the arguments in the right registers (a0-a5), so the stub only needs to load the system call number into a7 and execute ecall. No argument copying is needed. This elegant alignment between the C calling convention and the system call ABI is by design.

3.8 The complete system call path

To summarize, the complete path of a system call such as read(0, buf, 10) is:

  1. User program calls read(0, buf, 10) — the C compiler places arguments in a0=0, a1=buf, a2=10.
  2. The assembly stub loads a7=1 (SYS_read) and executes ecall.
  3. Hardware saves pc in mepc, sets mcause=11, jumps to mtvec.
  4. The assembly trap handler saves all registers to the trap frame, switches to the kernel stack, and calls c_trap_handler.
  5. c_trap_handler sees mcause=11 and calls syscall_dispatch.
  6. syscall_dispatch reads a7=1, calls sys_read, which reads data from the console device into buf.
  7. The result is stored in tf->a0; mepc is advanced by 4.
  8. trap_ret restores all registers from the trap frame and executes mret.
  9. The user program resumes at the instruction after ecall, with the result in a0.

3.9 Real world

The trap mechanism in logOS is simpler than in production systems because logOS runs entirely in machine mode (M-mode). Production RISC-V operating systems typically run the kernel in supervisor mode (S-mode) and user programs in user mode (U-mode). S-mode uses a different set of CSRs (stvec, sepc, scause, sscratch) and the sret instruction, but the mechanism is otherwise identical.

The xv6 teaching kernel uses a trampoline page mapped at the same virtual address in every process's page table, allowing the trap handler to work before page tables are switched. logOS does not need this because there are no page tables; all addresses are physical.

Linux's system call entry is far more complex, including optimizations like vDSO (virtual dynamic shared object) to handle some system calls entirely in user space without entering the kernel.

Chapter 4: Processes

A process is the operating system's abstraction for a running program. Each process has its own memory, its own registers (saved in a trap frame), its own file descriptors, its own environment variables, and a unique process identifier (PID). The kernel's process management code maintains these per-process resources and orchestrates the transitions between processes.

4.1 The process control block

The kernel represents each process with a process control block (PCB), defined as struct process (kernel/process.h): 

struct process {
    int              state;          /* PROC_FREE / PROC_RUNNING / PROC_READY */
    int              pid;            /* Process ID */
    int              parent;         /* Slot index of parent (-1 = kernel) */
    int              exit_code;      /* Exit status */
    trap_frame_t     tf;             /* Saved CPU registers (144 bytes) */
    uint32_t         mem_base;       /* Start of 64KB memory slot */
    uint32_t         stack_top;      /* Top of stack within slot */
    struct fd_entry  fds[MAX_FD];    /* Per-process file descriptors */
    char             env[MAX_ENVC][MAX_ENV_LEN];  /* Environment variables */
    int              env_count;      /* Number of env entries */
};

The global process table is an array of 8 such structures: 

struct process proc_table[MAX_PROCS];
int current_proc = -1;  /* Index of currently running process */

A process is in one of three states:

  • PROC_FREE (0) — The slot is available for use.
  • PROC_RUNNING (1) — The process is currently executing on the CPU.
  • PROC_READY (2) — The process is suspended, waiting for a child to finish.

Since logOS has no preemptive scheduling, at most one process is PROC_RUNNING at any time. Other processes are either PROC_FREE or PROC_READY (suspended parents).

4.2 Code: Process initialization

At boot, proc_init() (kernel/process.c) initializes the process table by computing each slot's memory addresses: 

void proc_init(void) {
    for (int i = 0; i < MAX_PROCS; i++) {
        proc_table[i].state = PROC_FREE;
        proc_table[i].pid = 0;
        proc_table[i].parent = -1;
        proc_table[i].mem_base = PROC_SLOT_BASE(i);
        proc_table[i].stack_top = PROC_SLOT_STACK(i);
    }
}

Process allocation scans for the first free slot: 

int proc_alloc(void) {
    for (int i = 0; i < MAX_PROCS; i++) {
        if (proc_table[i].state == PROC_FREE) {
            proc_table[i].pid = next_pid++;
            proc_table[i].state = PROC_READY;
            return i;
        }
    }
    return -1;  /* No free slot */
}

PIDs are assigned from a monotonically increasing counter. This means that after a process exits and its slot is reused, the new process gets a different PID. This is important: it prevents confusion between the old and new occupants of the same memory slot.

4.3 File descriptors

Each process has an array of 8 file descriptor entries. Each entry records whether it is in use, the inode number (for regular files), the current offset, and device information (for character devices) (kernel/syscall.h): 

struct fd_entry {
    int in_use;          /* 1 if this fd is open */
    uint32_t inode;      /* Inode number for files */
    uint32_t offset;     /* Current read/write position */
    uint8_t type;        /* FT_FILE or FT_CHARDEV */
    uint8_t major;       /* Major device number */
    uint8_t minor;       /* Minor device number */
};

When a process is created, proc_fd_init() opens the first three file descriptors on the console device: 

void proc_fd_init(int slot) {
    struct process *p = &proc_table[slot];
    for (int i = 0; i < MAX_FD; i++)
        p->fds[i].in_use = 0;

    /* fd 0 = stdin, fd 1 = stdout, fd 2 = stderr */
    for (int i = 0; i < 3; i++) {
        p->fds[i].in_use = 1;
        p->fds[i].type = FT_CHARDEV;
        p->fds[i].major = CONSOLE_MAJOR;
        p->fds[i].minor = 0;
    }
}

4.4 Code: spawn and exit

The sys_spawn system call (kernel/syscall.c) is the most complex system call in logOS. It must:

  1. Copy the path and arguments to kernel buffers (the caller's memory will remain intact, but the kernel needs its own copies for processing).
  2. Resolve the path to an absolute path.
  3. Allocate a new process slot for the child.
  4. Copy the parent's environment to the child.
  5. Load the ELF executable into the child's memory slot.
  6. Set up the child's stack with arguments.
  7. Set up the child's trap frame with the entry point, stack pointer, and argc/argv.
  8. Mark the parent as PROC_READY and the child as PROC_RUNNING.
  9. Switch to the child via trap_ret.

The child's stack layout after setup is: 

  stack_top
    argv string data   (argument strings copied from parent)
    [16-byte align]
    argv[0], argv[1], ..., NULL   (pointer array)
    [16-byte align]
    <- sp (child's initial stack pointer)

The spawn call is noteworthy for its use of trap_ret() to switch context. Rather than returning normally from the system call, sys_spawn calls trap_ret(&child->tf), which restores the child's registers and jumps to the child's entry point via mret. This is a non-local jump: sys_spawn never returns to its caller. Instead, when the child eventually calls exit(), sys_exit resumes the parent.

The sys_exit system call (kernel/syscall.c) performs the reverse operation. If the exiting process has a parent: 

static int sys_exit(trap_frame_t *tf) {
    struct process *cur = &proc_table[current_proc];
    int exit_status = (int)tf->a0;

    cur->exit_code = exit_status;

    if (cur->parent >= 0) {
        struct process *parent = &proc_table[cur->parent];
        parent->tf.a0 = (uint32_t)exit_status;  /* spawn's return value */
        parent->tf.mepc += 4;                    /* Skip past ecall */
        parent->state = PROC_RUNNING;
        proc_free(current_proc);
        current_proc = cur->parent;
        trap_ret(&parent->tf);  /* Resume parent — never returns */
    }

    return 1;  /* Top-level exit: signal kernel */
}

This is the crucial mechanism: sys_exit stores the child's exit code into the parent's a0 register (in the parent's trap frame), advances the parent's mepc past the ecall instruction, and then calls trap_ret with the parent's trap frame. The parent resumes execution as if spawn() had just returned, with the child's exit code as the return value.

4.5 Context switching without an MMU

In logOS, "context switching" is simpler than in systems with virtual memory. Since each process has its own fixed physical memory slot, switching between processes only requires saving and restoring registers (via the trap frame) and updating current_proc. There are no page tables to switch, no TLB to flush, and no address space to change.

The context switch happens in two places:

  • In sys_spawn: the kernel switches from parent to child by calling trap_ret(&child->tf).
  • In sys_exit: the kernel switches from child to parent by calling trap_ret(&parent->tf).

Both use the same mechanism: trap_ret restores a saved register set and executes mret. The difference is only which trap frame is restored.

4.6 Environment variables

Each process stores up to 16 environment variables, each up to 64 characters, in the format "NAME=value". The proc_env_copy function copies the parent's environment to the child during spawn: 

void proc_env_copy(int dst_slot, int src_slot) {
    struct process *dst = &proc_table[dst_slot];
    struct process *src = &proc_table[src_slot];
    for (int i = 0; i < src->env_count; i++) {
        strcpy(dst->env[i], src->env[i]);
    }
    dst->env_count = src->env_count;
}

Because the child gets a copy of the parent's environment, modifications made by the child (via setenv or unsetenv) do not affect the parent. This is the standard Unix semantics.

4.7 Real world

Production Unix systems manage processes very differently. The Linux kernel's task_struct is thousands of bytes and tracks hundreds of per-process attributes. Processes have their own virtual address spaces, signal masks, resource limits, cgroups, namespaces, and more.

The logOS process model most closely resembles early embedded operating systems or the MS-DOS "TSR" (terminate-and-stay-resident) programs, where a parent program loads a child and waits for it to complete. Modern preemptive systems allow multiple processes to be RUNNING simultaneously (on multi-core hardware) or interleaved via timer interrupts (on single-core hardware).

Chapter 5: The file system

The purpose of a file system is to organize and store data. File systems typically support sharing of data among users and applications, as well as persistence so that data is still available after a reboot.

The logOS file system provides Unix-like files, directories, and pathnames, and stores its data on a block device for persistence. The rest of this chapter explains how the file system is structured, from raw disk blocks up to named files and directories.

5.1 Overview

The logOS file system implementation is organized in five layers:

  1. The block layer reads and writes 512-byte blocks on the block device.
  2. The bitmap allocator tracks which blocks are free using a bitmap.
  3. The inode layer provides individual files, each represented as an inode with a unique inode number and some blocks holding the file's data.
  4. The directory layer implements each directory as a special kind of inode whose content is a sequence of directory entries, each containing a filename and inode number.
  5. The pathname layer resolves hierarchical path names like /bin/hello by recursively looking up each component.

5.2 Disk layout

The filesystem occupies 256 blocks of 512 bytes each (128KB total). The blocks are organized as follows: 

  Block 0          Superblock (filesystem metadata)
  Block 1          Block allocation bitmap
  Blocks 2-5       Inode table (32 inodes x 128 bytes = 8 per block)
  Blocks 6-255     Data blocks (for file/directory contents)
Figure 5.1: Filesystem disk layout.

The superblock (kernel/fs_types.h) records the filesystem's metadata: 

struct superblock {
    uint32_t magic;            /* FS_MAGIC (0x53465346 = "FSFS") */
    uint32_t total_blocks;     /* 256 */
    uint32_t total_inodes;     /* 32 */
    uint32_t free_blocks;      /* Number of free data blocks */
    uint32_t free_inodes;      /* Number of free inodes */
    uint32_t bitmap_start;     /* Block number of bitmap (1) */
    uint32_t bitmap_blocks;    /* Number of bitmap blocks (1) */
    uint32_t inode_start;      /* First block of inode table (2) */
    uint32_t inode_blocks;     /* Blocks in inode table (4) */
    uint32_t data_start;       /* First data block (6) */
};

The magic number is checked when mounting the filesystem to verify that the block device contains a valid filesystem. If the magic number doesn't match, the mount fails.

5.3 Code: Block layer

The block device is accessed through memory-mapped I/O registers at address 0x200000 (kernel/block.h): 

#define BLOCK_DEV_BASE     0x200000
#define BLOCK_CMD          (*(volatile uint32_t*)(BLOCK_DEV_BASE + 0x00))
#define BLOCK_MEM_ADDR     (*(volatile uint32_t*)(BLOCK_DEV_BASE + 0x04))
#define BLOCK_NUM          (*(volatile uint32_t*)(BLOCK_DEV_BASE + 0x08))
#define BLOCK_STATE        (*(volatile uint32_t*)(BLOCK_DEV_BASE + 0x0C))

To read a block, the kernel writes the destination memory address and block number to the MMIO registers, issues a read command, and busy-waits for completion (kernel/block.c): 

int block_read(uint32_t block_num, void *buf) {
    BLOCK_MEM_ADDR = (uint32_t)buf;
    BLOCK_NUM = block_num;
    BLOCK_CMD = CMD_READ;

    while (BLOCK_STATE == STATE_BUSY);  /* Wait for completion */

    if (BLOCK_STATE == STATE_ERROR) return FS_ERR_IO;
    return FS_OK;
}

This is essentially a DMA (Direct Memory Access) transfer: the block device reads data from the simulated disk directly into the specified memory buffer. The volatile keyword on the MMIO registers is essential; without it, the compiler might optimize away the busy-wait loop, since it doesn't see the hardware changing BLOCK_STATE.

5.4 Code: Block allocation bitmap

The bitmap tracks which blocks are in use. Each bit corresponds to one block: 1 means allocated, 0 means free. The bitmap occupies one block (512 bytes = 4096 bits), which is enough for up to 4096 blocks. 

int block_alloc(void) {
    if (sb.free_blocks == 0) return FS_ERR_NO_SPACE;

    for (uint32_t b = sb.data_start; b < sb.total_blocks; b++) {
        if (bitmap_get(b) == 0) {
            bitmap_set(b, 1);       /* Mark as used */
            sb.free_blocks--;
            block_write(0, &sb);    /* Update superblock */
            memset(block_buf, 0, BLOCK_SIZE);
            block_write(b, block_buf);  /* Zero the new block */
            return b;
        }
    }
    return FS_ERR_NO_SPACE;
}

The allocator always zeros a newly allocated block. This prevents stale data from a previously deleted file from appearing in a new file.

5.5 Code: Inodes

An inode (index node) describes a single file or directory. Each inode is 128 bytes (kernel/fs_types.h): 

struct inode {
    uint32_t size;                     /* File size in bytes */
    uint8_t  type;                     /* FT_FILE, FT_DIR, or FT_CHARDEV */
    uint8_t  major;                    /* Device major number */
    uint8_t  minor;                    /* Device minor number */
    uint8_t  reserved;
    uint16_t blocks[DIRECT_BLOCKS];    /* 60 direct block pointers */
};

The inode stores the file's size, its type, and up to 60 direct block pointers. Each pointer is a 16-bit block number. With 512-byte blocks, a single file can hold up to 60 × 512 = 30,720 bytes. For logOS's 128KB filesystem, this is more than sufficient.

Note that logOS inodes have no indirect block pointers. Unix file systems use single, double, and triple indirect blocks to support large files. logOS's simplified design trades off maximum file size for implementation simplicity.

The inode table holds 32 inodes. With 4 inodes per block (128 bytes each in a 512-byte block), the table occupies 8 blocks. Inode 0 is always the root directory.

5.6 Code: Directories

A directory is an inode whose data blocks contain an array of directory entries. Each entry is 32 bytes (kernel/fs_types.h): 

struct dirent {
    uint32_t inode;              /* Inode number (0xFFFFFFFF = free) */
    char     name[MAX_FILENAME]; /* Filename (28 chars, null-terminated) */
};

A free entry is marked with the sentinel value 0xFFFFFFFF in the inode field. When looking up a name in a directory, the kernel reads each data block and scans the entries (kernel/dir.c): 

int dir_lookup(uint32_t dir_ino, const char *name, uint32_t *found_ino) {
    struct inode dir;
    inode_read(dir_ino, &dir);

    if (dir.type != FT_DIR) return FS_ERR_NOT_DIR;

    for (int b = 0; b < DIRECT_BLOCKS; b++) {
        if (dir.blocks[b] == 0) continue;
        block_read(dir.blocks[b], block_buf);

        struct dirent *de = (struct dirent *)block_buf;
        for (int i = 0; i < DIRENTS_PER_BLOCK; i++) {
            if (de[i].inode != DIRENT_FREE &&
                strcmp(de[i].name, name) == 0) {
                *found_ino = de[i].inode;
                return FS_OK;
            }
        }
    }
    return FS_ERR_NOT_FOUND;
}

Adding an entry first searches for a free slot in existing blocks. If none is found, it allocates a new block, initializes all entries as free, and uses the first slot. Removing an entry simply marks it as free by setting inode = DIRENT_FREE.

5.7 Code: Path resolution

The fs_open function (kernel/fs.c) resolves an absolute pathname to an inode number by walking the directory tree: 

int fs_open(const char *path, uint32_t *ino) {
    if (path[0] != '/') return FS_ERR_INVALID;

    uint32_t current_ino = ROOT_INODE;
    const char *p = path + 1;

    if (*p == '\0') { *ino = ROOT_INODE; return FS_OK; }

    while (*p != '\0') {
        /* Extract next component */
        char name[MAX_FILENAME];
        int i = 0;
        while (*p != '\0' && *p != '/') name[i++] = *p++;
        name[i] = '\0';
        while (*p == '/') p++;

        /* Look up in current directory */
        uint32_t found_ino;
        int result = dir_lookup(current_ino, name, &found_ino);
        if (result != FS_OK) return result;

        current_ino = found_ino;
    }

    *ino = current_ino;
    return FS_OK;
}

The algorithm starts at the root inode and processes each path component by looking it up in the current directory. For the path /bin/hello, it first looks up bin in the root directory to get its inode, then looks up hello in the bin directory.

Relative paths are handled by resolve_path() in (kernel/syscall.c), which prepends the current working directory before calling fs_open.

5.8 Code: Creating directories

The fs_mkdir function (kernel/dir.c) creates a new directory: 

int fs_mkdir(uint32_t parent_ino, const char *name) {
    int ino = inode_alloc();             /* Allocate new inode */

    struct inode in;
    memset(&in, 0, INODE_SIZE);
    in.type = FT_DIR;
    inode_write(ino, &in);

    dir_add(ino, ".", ino);              /* . points to self */
    dir_add(ino, "..", parent_ino);      /* .. points to parent */
    dir_add(parent_ino, name, ino);      /* Add to parent */

    return ino;
}

Every directory contains . and .. entries. The . entry points to the directory itself, and .. points to its parent. For the root directory, .. also points to the root. These entries allow programs to navigate the directory tree using relative paths like ../sibling.

5.9 Real world

The logOS filesystem is deliberately simple compared to real-world filesystems. Production filesystems must address several challenges that logOS ignores:

Crash recovery. If the system crashes during a multi-step operation (like creating a file, which requires updating both the inode table and the directory), the filesystem may be left in an inconsistent state. Real filesystems use journaling (ext4, NTFS) or copy-on-write (ZFS, btrfs) to ensure atomic updates. logOS has no crash recovery mechanism.

Buffer cache. Real kernels cache recently-read disk blocks in RAM to avoid repeated disk reads. logOS reads blocks directly from the device each time, using a single shared block_buf.

Large files. The ext4 filesystem uses extent trees to efficiently represent large files. logOS's 60 direct block pointers limit file size to about 30KB.

Concurrent access. In multi-process or multi-threaded systems, multiple processes may access the filesystem simultaneously, requiring locking. logOS's single-process execution model avoids this complexity.

Chapter 6: Device drivers

An operating system must interact with hardware devices: keyboards, displays, disks, network interfaces, and more. Each device has its own protocol for communication. The kernel abstracts these differences behind a uniform interface, so that user programs can use the same read and write system calls regardless of whether the underlying object is a file on disk or a hardware device.

logOS implements a simple character device driver framework. Character devices transfer data one byte at a time (unlike block devices, which transfer fixed-size blocks). The console — the user's keyboard and screen — is logOS's only character device.

6.1 The device driver interface

Each device driver provides a set of callback functions (kernel/device.h): 

struct device_ops {
    int (*open)(uint8_t minor);
    int (*close)(uint8_t minor);
    int (*read)(uint8_t minor, void *buf, uint32_t len);
    int (*write)(uint8_t minor, const void *buf, uint32_t len);
};

Drivers are registered by major number, an integer that identifies the driver. The minor number is passed to each callback to distinguish between multiple instances of the same type of device (e.g., multiple serial ports).

The device framework maintains a table of registered drivers and dispatches operations to the appropriate driver (kernel/device.c): 

static struct device_ops *device_table[MAX_DEVICES];

int device_register(uint8_t major, struct device_ops *ops) {
    if (major >= MAX_DEVICES) return FS_ERR_INVALID;
    if (device_table[major] != 0) return FS_ERR_EXISTS;
    device_table[major] = ops;
    return FS_OK;
}

int device_read(uint8_t major, uint8_t minor, void *buf, uint32_t len) {
    if (major >= MAX_DEVICES || device_table[major] == 0)
        return FS_ERR_NOT_FOUND;
    if (device_table[major]->read == 0) return FS_ERR_INVALID;
    return device_table[major]->read(minor, buf, len);
}

6.2 Code: The console driver

The console driver (kernel/console_dev.c) implements reading from the keyboard and writing to the display, both through memory-mapped I/O: 

/* Console MMIO addresses */
#define CONSOLE_DATA  (*(volatile uint32_t *)0xFFFF000C)  /* Output */
#define CONSOLE_RCR   (*(volatile uint32_t *)0xFFFF0004)  /* Input ready */
#define CONSOLE_RDR   (*(volatile uint32_t *)0xFFFF0008)  /* Input data */

Writing to the console sends characters to the display: 

static int console_write(uint8_t minor, const void *buf, uint32_t len) {
    const uint8_t *src = (const uint8_t *)buf;
    for (uint32_t i = 0; i < len; i++) {
        CONSOLE_DATA = src[i];
    }
    return len;
}

Reading from the console polls for keyboard input: 

static int console_read(uint8_t minor, void *buf, uint32_t len) {
    uint8_t *dst = (uint8_t *)buf;
    uint32_t i = 0;
    while (i < len) {
        while ((CONSOLE_RCR & 1) == 0) { /* busy wait */ }
        dst[i++] = (uint8_t)CONSOLE_RDR;
    }
    return i;
}

The CONSOLE_RCR register's bit 0 indicates whether a character is available. The driver busy-waits until a character arrives, then reads it from CONSOLE_RDR. This is polling — the CPU repeatedly checks the device status. The alternative, interrupt-driven I/O, would have the device signal the CPU when data is available, freeing the CPU to do other work. logOS uses polling for simplicity.

The driver registers itself during kernel initialization: 

static struct device_ops console_ops = {
    .open  = 0,
    .close = 0,
    .read  = console_read,
    .write = console_write
};

int console_dev_init(void) {
    return device_register(CONSOLE_MAJOR, &console_ops);
}

The .open and .close function pointers are NULL because the console doesn't need any per-open or per-close actions.

6.3 Connecting devices to the filesystem

Unix's "everything is a file" principle means devices are accessed through the filesystem. The mknod system call creates a device file — a special filesystem entry that stores a major/minor number pair instead of data: 

int fs_mknod(uint32_t dir_ino, const char *name,
             uint8_t major, uint8_t minor) {
    int ino = inode_alloc();

    struct inode in;
    memset(&in, 0, INODE_SIZE);
    in.type = FT_CHARDEV;
    in.major = major;
    in.minor = minor;
    inode_write(ino, &in);

    dir_add(dir_ino, name, ino);
    return ino;
}

When a process opens a device file, sys_open detects that the inode type is FT_CHARDEV and stores the major/minor numbers in the file descriptor. Subsequent read and write calls check the file descriptor type and dispatch to the device driver: 

static int32_t sys_read(trap_frame_t *tf) {
    struct fd_entry *fde = get_fd(tf->a0);

    if (fde->type == FT_CHARDEV) {
        return device_read(fde->major, fde->minor, buf, len);
    } else if (fde->type == FT_FILE) {
        return file_read(fde->inode, fde->offset, buf, len);
    }
}

6.4 Real world

The logOS device driver framework is a minimal version of Unix's character device interface. Linux's driver model is far more elaborate, with dynamic major/minor number allocation, sysfs for device information, udev for device management, and support for hundreds of device types.

Polling, as used by the console driver, wastes CPU cycles. Real systems use interrupts: the device sends a hardware signal to the CPU, which triggers a trap handler. The handler services the device and returns. This allows the CPU to execute other code while waiting for slow I/O. logOS could be extended to support interrupt-driven console I/O by enabling machine-mode external interrupts and registering a handler in the trap dispatcher.

Chapter 7: Loading programs

For the kernel to run a user program, it must read the program's executable file from the filesystem and place the code and data into a process's memory slot. logOS uses the ELF (Executable and Linkable Format) file format, the standard for Unix-like systems. This chapter describes the ELF format, the loader that reads it, and the challenge of making programs position-independent so they can run at any memory address.

7.1 The ELF format

An ELF file begins with a header that identifies the file format and describes the program's layout: 

/* ELF magic: 0x7F 'E' 'L' 'F' */
typedef struct {
    uint8_t  e_ident[16];     /* Magic number and format info */
    uint16_t e_type;          /* ET_EXEC (fixed) or ET_DYN (relocatable) */
    uint16_t e_machine;       /* EM_RISCV for RISC-V */
    uint32_t e_entry;         /* Entry point address */
    uint32_t e_phoff;         /* Offset of program header table */
    uint16_t e_phnum;         /* Number of program headers */
    /* ... */
} Elf32_Ehdr;

The program headers describe segments — contiguous regions of the file that should be loaded into memory: 

typedef struct {
    uint32_t p_type;          /* PT_LOAD for loadable segments */
    uint32_t p_offset;        /* Offset in file */
    uint32_t p_vaddr;         /* Virtual address (where to load) */
    uint32_t p_filesz;        /* Bytes in file (code + initialized data) */
    uint32_t p_memsz;         /* Bytes in memory (includes BSS) */
} Elf32_Phdr;

A typical program has two loadable segments: one for code (.text and .rodata) and one for data (.data and .bss). The .bss section contains uninitialized global variables; it occupies space in memory (p_memsz) but not in the file (p_filesz), so the loader must zero the difference.

7.2 Code: The ELF loader

The loader (kernel/loader.c) reads an ELF file from the filesystem and loads it at a specified address. The loading process has three phases:

Phase 1: Validation. The loader reads the ELF header and checks the magic number, architecture (RISC-V 32-bit), and executable type.

Phase 2: Loading segments. The loader iterates over the program headers. For each PT_LOAD segment, it reads the file data into memory and zeros the BSS portion: 

for (i = 0; i < ehdr.e_phnum; i++) {
    file_read(ino, ph_offset, &phdr, sizeof(Elf32_Phdr));

    if (phdr.p_type != PT_LOAD) continue;

    uint32_t seg_addr = phdr.p_vaddr + load_bias;

    /* Load segment data from file */
    if (phdr.p_filesz > 0) {
        file_load_direct(ino, phdr.p_offset, (void *)seg_addr, phdr.p_filesz);
    }

    /* Zero BSS portion */
    if (phdr.p_memsz > phdr.p_filesz) {
        memset((void *)(seg_addr + phdr.p_filesz), 0,
               phdr.p_memsz - phdr.p_filesz);
    }
}

The load_bias is the difference between the actual load address (the process slot base) and the address the program was linked at (typically 0 for PIE executables). This bias is added to every address in the program.

Phase 3: Relocations. For position-independent executables (type ET_DYN), the loader applies relocations to fix up addresses that depend on the load location.

7.3 Position-independent code

logOS user programs are compiled as position-independent executables (PIE) using the -fPIE flag. A PIE executable can be loaded at any address — crucial because each process slot is at a different address.

The RISC-V auipc (Add Upper Immediate to PC) instruction is the key to position-independent code. Instead of loading an absolute address, the compiler generates PC-relative references: 

auipc  a5, %pcrel_hi(.LC0)      # a5 = PC + high 20 bits of offset
addi   a5, a5, %pcrel_lo(.LC0)  # a5 = a5 + low 12 bits of offset

This computes the address of .LC0 relative to the current PC, which is correct regardless of where the code is loaded. No relocations are needed for these references.

7.4 The relocation problem

Not all code can use PC-relative addressing. When the compiler needs to store a pointer in a global data structure (e.g., a static array of string pointers), it generates a relocation entry. The linker is supposed to process these at link time, but for PIE executables, the final addresses aren't known until load time.

The logOS loader handles R_RISCV_RELATIVE relocations: 

if (type == R_RISCV_RELATIVE) {
    uint32_t *target = (uint32_t *)(rela->r_offset + load_bias);
    *target = load_bias + rela->r_addend;
}

This patches each address in the loaded program by adding the load bias.

7.5 Known PIE limitations

The bare-metal RISC-V toolchain (riscv64-elf-*) has a limitation: it does not support the -pie linker flag. This means some pointer relocations are resolved at link time and cannot be fixed at load time. This affects static pointer array initialization: 

/* BAD — pointers stored in .data use absolute addresses */
char *argv[] = { "hello", "arg1", NULL };

/* GOOD — build array at runtime using PC-relative references */
char *argv[3];
argv[0] = "hello";
argv[1] = "arg1";
argv[2] = NULL;

In the "bad" example, the compiler stores pointers to string literals in the .data section. These pointers are absolute addresses resolved at link time (typically address 0). When the program is loaded at 0x110000, the pointers still contain the old addresses — completely wrong.

In the "good" example, each assignment generates auipc/addi instructions that compute the string address relative to the current PC. This works correctly regardless of where the program is loaded.

Similarly, static or global arrays whose addresses are passed to system calls should be placed on the stack: 

/* BAD — compiler may use absolute address for static array */
static struct dirent entries[32];
readdir("/", entries, 32);

/* GOOD — stack address is always correct */
struct dirent entries[32];
readdir("/", entries, 32);

7.6 Real world

The ELF format is used by virtually all Unix-like systems. Production ELF loaders are more complex: they handle shared libraries, symbol resolution, lazy binding via the PLT (Procedure Linkage Table) and GOT (Global Offset Table), thread-local storage, and dozens of relocation types. The ld-linux.so dynamic linker on Linux is itself an ELF executable.

Modern systems have full PIE support in their toolchains, and security features like ASLR (Address Space Layout Randomization) depend on it. The toolchain limitation that logOS encounters is specific to the bare-metal riscv64-elf toolchain, which disables PIE support because newlib (the embedded C library) doesn't support shared libraries.

Chapter 8: The shell

The shell is the user's primary interface to the operating system. It reads commands typed at the keyboard, interprets them, and executes them. Despite being central to the user experience, the shell is not part of the kernel. It is an ordinary user program that exercises the system call interface. This is an important design principle: the power of the system call interface should be sufficient to build a fully functional command interpreter without any special kernel support.

8.1 Structure of the shell

The logOS shell (user/shell.c) follows a classic read-eval-print loop (REPL):

  1. Print a prompt showing the current working directory.
  2. Read a line of input from the console.
  3. Expand $VAR references in the input.
  4. Parse the line into an argument list (argc/argv).
  5. Execute the command: either a built-in or an external program.
  6. Repeat.

The main loop is straightforward: 

for (;;) {
    char *cwd = getenv("CWD");
    if (cwd) puts(cwd);
    puts("$ ");
    gets(line, MAX_CMD_LEN);
    if (line[0] == '\0') continue;

    expand_vars(line, expanded, MAX_CMD_LEN);
    cmd_argc = parse_cmd(expanded, cmd_argv, MAX_ARGS);
    if (cmd_argc == 0) continue;

    if (strcmp(cmd_argv[0], "help") == 0) cmd_help();
    else if (strcmp(cmd_argv[0], "exit") == 0) exit(code);
    else if (strcmp(cmd_argv[0], "cd") == 0) cmd_cd(argc, argv);
    else run_external(cmd_argc, cmd_argv);
}

8.2 Variable expansion

Before parsing, the shell expands environment variable references in the input. The expand_vars function handles three forms:

  • $NAME — expands to the value of the environment variable NAME.
  • ${NAME} — explicit braces for disambiguation.
  • $? — the exit code of the last program.

For example, if PATH=/bin: 

$ echo $PATH        # prints: /bin
$ echo ${PATH}x     # prints: /binx
$ hello
$ echo $?           # prints: 42 (hello's exit code)

8.3 Built-in commands

Some commands must be built into the shell because they affect the shell's own state:

  • cd [dir] — changes the shell's current working directory via chdir(). An external program cannot change the shell's CWD.
  • set [VAR=value] — sets an environment variable in the shell's own environment, or lists all variables if called with no arguments.
  • unset VAR — removes an environment variable.
  • echo [args] — prints its arguments (expanded variables).
  • exit [code] — terminates the shell.
  • help — lists available commands.

The reason cd must be built-in is subtle. If cd were an external program, it would run in a child process. The child's chdir() call would change the child's CWD, but when the child exits, the shell's CWD would be unchanged (since environment changes in children don't affect parents). This is the same reason cd is a built-in in every Unix shell.

8.4 External commands and PATH lookup

Commands that are not built-in are run as external programs. The shell uses the PATH environment variable to locate executables: 

static void run_external(int argc, char *argv[])
{
    if (strchr(argv[0], '/') != NULL) {
        /* Path contains '/' — use as-is */
        result = spawn(argv[0], argv);
    } else {
        /* Search PATH directories */
        char *path_var = getenv("PATH");
        /* Walk colon-separated PATH entries, try each:
         *   path = "dir/cmd"
         *   result = spawn(path, argv);
         */
    }
}

If the command contains a /, it is treated as a path and used directly. Otherwise, the shell tries each directory in PATH in order. For example, with PATH=/bin, typing hello causes the shell to try spawn("/bin/hello", argv).

The spawn() call blocks until the child exits and returns the child's exit code. The shell stores this in the ? environment variable so the user can inspect it with echo $?.

8.5 Real world

Real Unix shells (bash, zsh, fish) are far more capable. They support:

  • Pipelines: ls | grep foo | wc -l
  • I/O redirection: echo hello > file.txt
  • Background execution: sleep 10 &
  • Job control: Ctrl-Z to suspend, fg/bg
  • Scripting: loops, conditionals, functions
  • Quoting: single quotes, double quotes, escaping
  • Globbing: *.c expanding to matching filenames
  • Tab completion: completing filenames and commands

The logOS shell supports none of these, but the simplicity makes it easy to understand the fundamental loop: read, parse, execute.

Chapter 9: Summary

This text introduced the main ideas in operating systems by studying one operating system, logOS, line by line. Some code lines embody the essence of the main ideas — the csrrw instruction that atomically swaps registers to enter the trap handler, the mret instruction that returns to user space, the loop in dir_lookup that searches for a filename, the trap_ret call in sys_exit that resumes a parent process. Each of these captures a fundamental operating system mechanism in a few lines of code.

Other code lines provide an illustration of how to implement a particular operating system idea and could easily be done differently. A better filesystem might use B-trees for directory lookup. A better process model might support preemptive scheduling. A better memory model might use virtual address spaces with hardware protection. These are all important topics that logOS deliberately omits in order to focus on the essentials.

The ideas explored in this text were:

  • System calls as the interface between user programs and the kernel, mediated by the RISC-V ecall instruction and trap machinery.
  • Processes as the kernel's abstraction for running programs, each with its own memory, file descriptors, environment, and saved register state.
  • The file system as a tree of named objects built from blocks, inodes, and directory entries — a hierarchy that turns raw disk storage into something meaningful.
  • Device drivers as the mechanism that unifies hardware access behind the same read/write interface used for files.
  • Program loading as the bridge between files on disk and running processes, requiring an understanding of executable formats and position-independent code.
  • The shell as a demonstration that powerful functionality can be built entirely in user space, using only the system call interface.

All the ideas were illustrated in the context of one particular, very successful system call interface: the Unix interface. But those ideas carry over to the design of other operating systems. Understanding how logOS works — from the first instruction after power-on to the shell prompt — provides a foundation for studying any operating system, from embedded systems to the world's largest servers.
